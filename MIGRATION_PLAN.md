# 旅行助手重构方案（Hello Agent -> LangChain + LangGraph）

> 目标：在可学习、可验收、可回滚的前提下，完成以下能力建设：
> - LangChain / LangGraph
> - ReAct
> - RAG 知识库能力
> - Skills（业务技能）
> - 长短期记忆
> - 经验复用
> - MCP 工具接入

## 1. 设计原则

1. **组合式里程碑**：每个里程碑可一次引入多项能力，但每项能力都要有独立验收。
2. **双轨运行**：重构期间保留 `legacy` 与 `next` 两套入口，便于回归比较。
3. **先观测后优化**：每个节点要有日志、耗时、token、工具调用记录。
4. **先框架后数据**：先把 LangGraph 流程跑通，再填充 RAG 与记忆数据。
5. **可回滚**：每次只在 feature flag 下开启新能力。

---

## 2. 推荐顺序（非线性、可组合）

你提出“不必按顺序、可一次加多个”，所以推荐以下 **4 个组合阶段**：

### 阶段 A：基础骨架 + ReAct（组合）

**目标**：快速得到“可思考、可调用工具”的新架构雏形。

**一次引入的能力**：
- LangChain（模型/Prompt/工具适配层）
- LangGraph（状态机编排）
- ReAct（推理-行动循环）

**为什么先组合这三个**：
- ReAct 在 LangGraph 里落地最自然；
- 先把 agent 决策骨架稳定下来，后续加 RAG/记忆只是在图上加节点。

**建议流程图（MVP）**：
- `START`
- `Intent Router`（用户意图分类：闲聊/规划/问答）
- `Planner(ReAct Thought)`
- `Tool Executor(Action -> Observation)`
- `Answer Synthesizer`
- `END`

**实现细节**：
- 状态对象统一为 `AgentState`：
  - `user_query`
  - `conversation_context`
  - `tool_calls[]`
  - `retrieval_chunks[]`
  - `memory_hits[]`
  - `final_answer`
- ReAct 限制：
  - `max_iterations = 4~6`
  - 每步 action 参数必须通过 schema 校验
  - 出错进入 `fallback_node`
- 观测：
  - 每个 node 输出输入摘要、耗时、token
  - ReAct 每一步记录 `thought/action/observation`

**验收标准**：
- 复杂问题可触发多步工具调用；
- 工具失败不崩溃，能回退；
- 全链路日志可读。

---

### 阶段 B：RAG + Skills（组合）

**目标**：在“会行动”的基础上，做到“有知识依据 + 能力模块化”。

**一次引入的能力**：
- RAG（知识库构建与检索）
- Skills（可插拔业务能力）

**为什么这两个要组合**：
- 很多 skill（如签证建议、路线建议）本身依赖检索结果；
- skill 输出可与 RAG 引文绑定，提升可信度。

**实现细节：RAG**
- 文档源：
  - 城市攻略、景点资料、交通规则、签证政策、内部 FAQ
- 数据处理：
  - 清洗 -> 分块（按语义段落 + 重叠窗口）-> embedding -> 向量索引
- 检索策略：
  - Query Rewrite（可选）
  - Hybrid Search（向量 + 关键词，可后续演进）
  - Top-k + rerank
- 生成策略：
  - `Answer with citations`
  - 不命中时显式说明“知识库暂无依据”

**实现细节：Skills**
- 每个 skill 采用统一契约：
  - `name`
  - `input_schema`
  - `execute()`
  - `output_schema`
  - `failure_strategy`
- 首批建议 skills：
  - `itinerary_planner`
  - `budget_optimizer`
  - `transport_recommender`
  - `visa_policy_checker`
- 在图中增加 `Skill Router`：根据意图 + 约束 + 检索命中选择 skill。

**验收标准**：
- 答案可附来源片段；
- 新增一个 skill 不需改核心图结构；
- 对知识型问题正确率显著提升（与无 RAG 基线对比）。

---

### 阶段 C：长短期记忆 + 经验复用（组合）

**目标**：让助手“记住用户”并“复用过往成功案例”。

**一次引入的能力**：
- 短期记忆（会话级）
- 长期记忆（用户级）
- 经验复用（案例级）

**为什么这三个要组合**：
- 三者都属于“历史信息利用”，可统一存储抽象，避免后续重复改造。

**实现细节：短期记忆**
- 采用 rolling summary：
  - 超过窗口时自动摘要
  - 保留硬约束（预算、天数、出行人群）

**实现细节：长期记忆**
- 仅存稳定偏好：
  - 出行风格（慢游/高密度）
  - 预算区间
  - 酒店偏好
- 写入策略：
  - 需“重复出现”或“用户明确确认”才写入
- 读取策略：
  - 生成前注入偏好；若与当前指令冲突，以当前指令优先

**实现细节：经验复用**
- 经验单元（Case）结构：
  - 问题画像（城市/时长/预算/人群）
  - 成功计划
  - 用户反馈分
- 推理时检索相似案例，作为 planner 的参考上下文（非硬约束）。

**验收标准**：
- 跨会话能延续偏好；
- 用户临时变更偏好可覆盖历史；
- 相似问题下规划稳定性提升。

---

### 阶段 D：MCP 接入 + 生产化治理（组合）

**目标**：接入真实外部能力，并增强可靠性。

**一次引入的能力**：
- MCP（标准工具协议）
- 生产治理（限流、超时、熔断、审计）

**建议接入顺序**：
1. 天气 MCP（低风险）
2. 地图/POI MCP
3. 票务/酒店 MCP（高风险，需更严格审计）

**实现细节**：
- MCP 适配层统一超时与错误码映射
- 工具健康检查 + 降级策略：
  - 不可用时提供“离线建议 + 明确声明”
- 安全与成本：
  - 工具白名单
  - 每轮最多工具调用数
  - 敏感参数过滤

**验收标准**：
- 外部工具异常时整体可用；
- 调用链路可审计；
- 成本与延迟可控。

## 3. 你可以采用的 3 种推进方式

1. **稳健学习型（推荐）**：A -> B -> C -> D
2. **RAG 优先型**：A（精简）-> B（重点）-> C -> D
3. **工程落地型**：A -> D（先打通外部能力）-> B -> C

---

## 4. 每阶段的统一“可学习验收清单”

- 功能：本阶段目标是否可演示
- 质量：是否有最小测试（单测/集成）
- 可观测：是否可看到 graph 节点轨迹
- 可回滚：feature flag 一键关闭
- 对比：与 legacy 的输出差异是否可解释

## 5. 最小目录建议（便于后续编码）

- `app/graph/`：LangGraph 节点与边定义
- `app/agents/`：ReAct planner/executor
- `app/skills/`：技能注册与实现
- `app/rag/`：索引构建、检索、重排
- `app/memory/`：短期/长期/案例记忆
- `app/mcp/`：MCP client 与 tool adapter
- `app/observability/`：日志、trace、metrics
- `evals/`：离线评测集与脚本

## 6. 风险与规避

- **风险 1：一次改太多无法定位问题**
  - 规避：组合阶段内部也要 feature flag 分开
- **风险 2：RAG 命中率低导致回答漂移**
  - 规避：先做检索评测，再调 chunk 和 rerank
- **风险 3：记忆污染（错误偏好写入）**
  - 规避：写入阈值 + 用户确认机制
- **风险 4：MCP 不稳定拖垮主流程**
  - 规避：超时、重试、熔断、降级模板

## 8. 修改点在项目功能中的体现（避免“为加功能而加功能”）

> 这一节回答“每个改动到底改善了什么用户体验/业务能力”。

### 8.1 LangGraph（替换单链路）
- **修改点**：把原本线性处理改为显式状态图（路由、规划、工具、汇总、回退）。
- **功能体现**：
  - 用户问“先给我3天上海亲子游，再改成预算3000”，系统能稳定进入“规划 -> 约束重算 -> 输出”，而不是上下文混乱。
  - 某个工具报错时可走 fallback，不会整轮失败。
- **业务价值**：可维护、可调试，后续加能力不破坏主流程。

### 8.2 ReAct（推理-行动循环）
- **修改点**：从一次性回答升级为“思考 -> 调工具 -> 观察 -> 再思考”。
- **功能体现**：
  - 对“帮我比较虹桥和浦东落地后的交通+酒店片区建议”这类复杂问题，会主动分步查询并综合，而不是拍脑袋。
  - 结果更有“推导过程”，错误更容易定位。
- **业务价值**：复杂任务成功率提升，工具调用更可控。

### 8.3 RAG（知识库检索增强）
- **修改点**：把旅游政策/攻略/FAQ 纳入可检索知识库，并在回答时引用来源。
- **功能体现**：
  - 用户问“某城市地铁末班时间/景点闭馆日/签证材料”，回答可给依据片段，而非模型记忆猜测。
  - 知识更新后无需重新训练模型，重建索引即可生效。
- **业务价值**：准确性与可信度提升，支持“可追溯回答”。

### 8.4 Skills（能力模块化）
- **修改点**：把“行程规划、预算优化、交通建议、签证核查”等拆成独立 skill。
- **功能体现**：
  - 用户一句“预算砍20%但不降体验”，可直接命中 `budget_optimizer` 而不是全局大模型重写。
  - 新增“露营线路推荐”只需加 skill，不必重构主 agent。
- **业务价值**：能力复用快、扩展成本低、回归范围小。

### 8.5 长短期记忆
- **修改点**：分离会话短期记忆与用户长期偏好记忆。
- **功能体现**：
  - **短期**：在当前会话中记住“带老人、不要早起、酒店要近地铁”。
  - **长期**：跨会话保留“用户偏好舒适型酒店、预算区间、饮食偏好”。
  - 用户临时改口“这次特种兵玩法”，系统能覆盖历史偏好。
- **业务价值**：个性化更稳定，减少重复沟通成本。

### 8.6 经验复用（Case Reuse）
- **修改点**：沉淀历史成功案例，按相似条件检索复用。
- **功能体现**：
  - 面对“周末亲子杭州2天预算2k”可参考历史高评分方案并快速产出。
  - 方案质量不再完全依赖当次随机生成。
- **业务价值**：稳定性提升，优质方案可规模化复用。

### 8.7 MCP（外部工具协议接入）
- **修改点**：通过 MCP 标准化对接天气、地图、POI、票务等外部服务。
- **功能体现**：
  - 用户问“明天下雨还适合去XX吗”，可实时调用天气+室内备选推荐。
  - 工具不可用时给降级答案，不中断对话。
- **业务价值**：从“建议型助手”升级为“可执行、可连接外部世界”的助手。

### 8.8 观测与治理（不是附属项，而是功能保障）
- **修改点**：增加日志、链路追踪、超时、限流、重试、熔断。
- **功能体现**：
  - 能解释“为什么这次推荐变慢/变差”。
  - 高峰时段仍可用，不会因为单一工具波动导致服务雪崩。
- **业务价值**：上线可运维、可审计、可持续优化。

### 8.9 你能直接感知到的最终变化（用户视角）
- 回答更“像顾问”：会分步骤思考并给依据。
- 方案更“像你定制的”：记住偏好，能延续历史。
- 结果更“能落地”：可调用外部工具拿到实时信息。
- 出错更“有兜底”：失败可回退，不是一报错就结束。
